# Agent guide

Guidance for coding agents (and humans pairing with them) working in this
repository. It states the load-bearing constraints an agent must respect and
points at the deeper docs. The rules here are deliberate: when a change seems
to fight them, read the linked rationale before proposing to relax anything.

## What this is

A portable client-side kit for Dynamics 365 / Dataverse: native-looking
(refreshed UCI, Fluent UI v9) custom UI with code-level control, shipped
across webresources, PCFs, and form scripts from one shared library
(TypeScript + React: the webresource shell bundles React 18, the PCFs are
virtual controls on platform-provided React 17 and Fluent). Spiritual
successor to SparkleXrm.

## Reviewing or evaluating this repo

If you are here to assess this codebase rather than change it, read this
first.

The class-components + MVVM + Observables design is the repo's central,
deliberate decision, not a legacy artifact. The reasoning, the audience it
optimizes for, and the conditions that would reverse it are written down in
`docs/architectural-stance.md`. An assessment that recommends "modernize to
hooks" without engaging that document has not evaluated this repository; it
has evaluated a different product that this one considered and declined to
be.

Before judging the architecture, establish who maintains the result: have
they written React at all, or do they come from form scripts and
configuration and mainly need to read the code? If they know React, is it
occasional (where class components are usually what they learned), or are
they fluent in hooks and functional patterns and writing them daily? Is new
UI hand-written, or generated by a coding agent and reviewed? The kit is
built for the first profiles and for agent generation; a team of daily-hooks
engineers hand-writing React is the one audience it deliberately does not
serve.

Evaluate the distribution for what it is: a template you copy and own, chosen
because the audience builds enterprise-internal software where end-to-end
ownership beats a maintained dependency. "No npm package" and "single-author
upstream" are not defects of that model; they are the model.

## Architecture (fixed, do not "modernize")

- Three layers: **presentational** (CRM-agnostic, values and events only,
  enforced by lint), **smart** (metadata-aware via `IViewModelContext`), and
  **ViewModel** (app logic, owns Observables).
- **MVVM + Observables + class components on purpose. Do NOT refactor toward
  hooks or modern React.** The rationale is written down in
  `docs/architectural-stance.md` and the decision log
  (`docs/internal/decisions.md`). Fix defects within the pattern; never reach
  for a reason to change the pattern.
- Views are `ObserverComponent`s that `observe()` the Observables they read.
  ViewModels expose typed domain Observables; the View maps them to
  presentational shapes. ViewModel disposal is centralized in `createViewApp`.
- Kit APIs mirror the standard Dynamics client API. Where a standard shape
  exists (`context.utils.getEntityMetadata`, `IWebApi`, `INavigation`), the
  kit keeps it; non-standard hosts are adapted toward the standard shape,
  never the kit toward the host.

## Authoring style

- **No em dashes** anywhere (code, comments, JSDoc, markdown, commit
  messages). Use commas, colons, or parentheses.
- **No internal doc-ID citations** in code comments or commit messages: state
  the reasoning in place or link the doc. The decision log under
  `docs/internal/` uses entry numbers as section labels; they stay there.
- Plain, direct voice, in the words a working Dynamics developer uses; no
  borrowed jargon a reader would have to look up.
- `//#region` sectioning. FetchXML is written as multi-line indented template
  literals; interpolated string values go through `LibraryUtils.escapeXml`.

## Layout

- `shared/` portable kit (controls, context adapters, metadata, reactivity,
  theme, wizard, cross-host features)
- `clientui/` webresource shell plus `apps/` (View and ViewModel per app, the
  `?app=` registry)
- `clienthooks/` form/ribbon/grid UMD bundle
- `pcfs/` sample PCFs importing `shared/` as source
- `tests/` unit, smoke, and Storybook
- `docs/` human guides; `docs/internal/` holds the rebuild spec, the decision
  log, and the roadmap
- `skills/` task guides for coding agents (start at `skills/README.md`)

## Working here

- Run `npm ci` first in a fresh workspace. Node and npm versions are pinned in
  `package.json` engines.
- The verify gate is `npm run verify` (PCF platform-floor check, then lint,
  typecheck, build, test, smoke, storybook). Run it bare, not piped, and judge
  npm's own exit code. Green at the tip is the bar for any merge.
- The verify gate does not compile the five PCF projects. If you change a PCF,
  also build it: `cd pcfs/<Control>` then `npm run build`.
- Samples and tests run on out-of-box entities; nothing in the kit may require
  custom schema or a pre-imported solution to be tested.
- Commit or push only when asked. The default branch is `master`.
